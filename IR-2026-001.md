## 1. Executive Summary

On 2026-06-22, a brute-force authentication attack was detected against an internal Windows 10 workstation (DESKTOP-HBCQGNE). The attack originated from host 192.168.1.14 (a Kali Linux machine) and targeted the local user account "Tsubasa" over the SMB protocol (port 445).

The attacker attempted multiple passwords in a short period of time. All attempts failed and no unauthorized access was gained. After repeated failed attempts, the Windows account lockout policy automatically engaged and blocked the account, stopping the attack.

Wazuh SIEM detected the activity and raised a high-severity correlation alert (rule 60204, level 10), correctly classifying it as a brute-force attack. The incident is considered resolved.

## 4. Technical Analysis

The attacker at 192.168.1.14 conducted a password-guessing attack against the "Tsubasa" account using the SMB protocol. Analysis of the Windows Security events (Event ID 4625) revealed the following:

- **Logon Type 3** confirms the attempts arrived over the network (remote), consistent with a tool-based SMB brute-force rather than interactive console access.
- **Sub-status 0xC000006A** ("bad password") indicates the username "Tsubasa" is valid; only the password was incorrect. This shows the attacker had a valid username and was attempting to guess the password.
- **NTLM** authentication package was used, typical of SMB-based credential attacks.
- A total of **19+ failed attempts** were recorded in two waves (~05:35 and ~05:47 UTC).

Wazuh logged each individual attempt under rule 60122 (level 5). When the failures accumulated, the correlation rule 60204 (level 10) aggregated 8+ failures and correctly classified the activity as Brute Force (MITRE T1110, Credential Access).

---

## 5. Detection

The attack was detected by **Wazuh SIEM**.

- **Individual events:** Event ID 4625 (failed logon), rule 60122, level 5.
- **Correlation alert:** rule 60204 ("Multiple Windows Logon Failures"), level 10.
- **MITRE mapping:** T1110 (Brute Force) / Tactic: Credential Access.

The analyst identified the alert by filtering for high-severity events (`rule.level >= 10`), which separated the real attack (level 10 correlation alert) from low-level noise (level 5 individual events).

*Note: The individual rule (60122) mislabeled the technique as T1531 (Account Access Removal). The correlation rule (60204) correctly identified it as T1110 (Brute Force). The analyst confirmed T1110 as the accurate classification.*

---

## 6. Response & Mitigation

- The **Windows account lockout policy** automatically engaged after multiple failed login attempts in a short time, blocking the "Tsubasa" account and stopping the attack.
- All login attempts failed (status 0xC000006D); no unauthorized access was gained.
- The source host 192.168.1.14 was identified for follow-up.

## 8. Recommendations

1. Ensure the "Tsubasa" account uses a strong, complex password.
2. Block the source IP (192.168.1.14) at the network / firewall level.
3. Keep the account lockout policy enabled (it successfully stopped this attack).
4. Close SMB (port 445) on this host if it is not required.
5. Monitor 192.168.1.14 for any further activity or new attacks.
